**transfer-orchestrator.ts**: Core server-side transfer logic. Exports:

1. `orchestrateTransfer(params)` -- Called asynchronously (fire-and-forget) from the human_detected webhook handler. Params: `{ callId: string, vapiCallId: string, vapiControlUrl: string, userPhone: string }`.
   - Update call status to `transferring`, log `transfer_initiated` event, push Pusher `status-change` with `transferring`.
   - Create unique conference name: `transfer-${callId}`.
   - Call the user via `twilioClient.conferences(conferenceName).participants.create()` with:
     - `from`: `process.env.TWILIO_PHONE_NUMBER!` (the Twilio number used for OTP/general)
     - `to`: `userPhone`
     - `startConferenceOnEnter: true`
     - `endConferenceOnExit: true` (end conference when user hangs up)
     - `beep: false` (silent -- user hears nothing, per locked decision)
     - `statusCallback`: `${process.env.NEXT_PUBLIC_APP_URL}/api/webhooks/twilio/conference-status`
     - `statusCallbackEvent`: `["initiated", "ringing", "answered", "completed"]`
   - Store the user's `participant.callSid` in the call event metadata for reference.
   - On Twilio error: call `handleTransferFailure(callId, vapiControlUrl, "bridge_failed")`.

2. `handleUserAnswered(callId)` -- Called by conference-status webhook when user picks up.
   - Fetch the call record from DB (need `vapiCallId`, `vapiControlUrl`).
   - Fetch the Vapi call via `vapiClient.calls.get(vapiCallId)` to get `phoneCallProviderId` (Twilio CallSid for the ISP call).
   - **Approach B (primary):** Fetch the Vapi call via `vapiClient.calls.get(vapiCallId)` to get `phoneCallProviderId` (Twilio CallSid). Then redirect the ISP call directly into the conference: `twilioClient.calls(twilioCallSid).update({ twiml: '<Response><Dial><Conference ...>transfer-{callId}</Conference></Dial></Response>' })`. This is the most direct approach -- no bridge number needed.
   - **Approach A (fallback, if `phoneCallProviderId` is unavailable):** Use Vapi controlUrl transfer to send the ISP call to a bridge number: `fetch(controlUrl, { type: "transfer", destination: { type: "number", number: process.env.TWILIO_CONFERENCE_BRIDGE_NUMBER! } })`. The bridge number answers with TwiML routing to the conference. **Lookup mechanism:** When `orchestrateTransfer` starts, store the mapping `{ ispTwilioCallSid → callId }` in the call record by updating a new `phoneCallProviderId` column -- OR fetch it early and store it. The `conference-bridge/route.ts` endpoint receives the `CallSid` in Twilio's form data and queries the DB: `SELECT id FROM call WHERE phone_call_provider_id = ?` to determine which conference to join.
   - After ISP call is redirected to conference: drop the AI leg via `fetch(controlUrl, { type: "end-call" })`.
   - Log `user_connected` and `ai_dropped` events.
   - Update status to `connected`, push Pusher `status-change` with `connected`.

3. `handleUserNoAnswer(callId)` -- Called by conference-status webhook when user doesn't answer.
   - Fetch call record (need `vapiControlUrl`).
   - Use Vapi controlUrl `say` action: `{ type: "say", content: "I'm sorry, they're unavailable right now. We'll call back later. Thank you!", endCallAfterSpoken: true }` (exact wording per locked decision).
   - Update status to `failed`, log `failed` event with reason `user_no_answer`.
   - Push Pusher `status-change` with `failed` and `call-ended` with `endedReason: "user-no-answer"`.

4. `handleConferenceEnded(callId)` -- Called by conference-status webhook when conference ends.
   - Update status to `completed`, set `endedAt`, log `completed` event.
   - Push Pusher `call-ended`.

5. `handleTransferFailure(callId, controlUrl, reason)` -- Generic failure handler.
   - Use controlUrl `say` with apology + `endCallAfterSpoken: true`.
   - Update status to `failed`, log event with reason.
   - Push Pusher events.

Import `db`, `call`, `callEvent` from schema. Import `pusherServer` and `callChannel`. Import `twilioClient` from new module. Import `vapiClient` from existing module.

**conference-status/route.ts**: Twilio conference status callback webhook. POST handler that:
- Parses `req.formData()` (Twilio sends form-encoded, NOT JSON).
- Extracts: `CallStatus`, `ConferenceSid`, `FriendlyName`, `CallSid`.
- Derives `callId` from `FriendlyName` by removing `transfer-` prefix.
- Routes based on `CallStatus`:
  - `"in-progress"` (Twilio sends this when participant answers): call `handleUserAnswered(callId)`.
  - `"no-answer"`, `"busy"`, `"failed"`, `"canceled"`: call `handleUserNoAnswer(callId)`.
  - `"completed"`: call `handleConferenceEnded(callId)`.
- Returns `new Response("OK", { status: 200 })` always.
- **IMPORTANT:** Twilio statusCallback for participants sends `CallStatus` for the participant's call leg status (initiated, ringing, in-progress, completed), NOT the conference event. The `in-progress` status means the user answered. The `completed` status means the participant's leg ended. Distinguish between "user leg completed" (could mean user hung up from the conference) and other events.

**conference-bridge/route.ts**: TwiML endpoint for Approach A fallback. POST handler that:
- Receives a Twilio request when the ISP call is transferred to the bridge number.
- Looks up the callId from the incoming `CallSid` (the ISP call's Twilio SID) -- uses a lookup against the call record where we stored the `phoneCallProviderId`.
- Returns TwiML: `<Response><Dial><Conference beep="false" startConferenceOnEnter="true" endConferenceOnExit="false">transfer-{callId}</Conference></Dial></Response>`.
- Set `Content-Type: text/xml`.
- This is the Approach A fallback -- may not be needed if Approach B works, but implement it for robustness.
