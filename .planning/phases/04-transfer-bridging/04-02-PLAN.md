  <name>Task 1: Validate user phone number in call start route</name>
  <files>
    src/app/api/call/start/route.ts
  </files>
  <action>
The transfer orchestrator needs the user's phone number to call them back. Currently, the phone number is on the `user` table but not on the `call` record. Two options: (A) join to user table at transfer time, or (B) denormalize onto the call record. Use option A (join) to avoid a migration -- the user's phone is already in the DB via `user.phoneNumber`.

**call/start/route.ts changes:**
No schema migration needed. The orchestrator will look up the user's phone from the `user` table via the `call.userId` foreign key. However, we need to ensure the phone number is available. Add a check in the call start route: if `session.user.phoneNumber` is falsy, return a 400 error ("Phone number required for callback"). This is a safety check -- all users should have a phone number via SMS auth.

No changes to `schema.ts` needed for this task -- the `user.phoneNumber` column already exists.

**Verify** the session object includes `phoneNumber` by checking the BetterAuth session type. If `phoneNumber` is not on `session.user`, query the user table directly: `await db.query.user.findFirst({ where: eq(user.id, session.user.id) })`.

Add the phone validation after the existing session check:
```typescript
// After session check, before ISP lookup
const userPhone = session.user.phoneNumber;
if (!userPhone) {
  return NextResponse.json(
    { error: "Phone number required for callback" },
    { status: 400 }
  );
}
```

If `phoneNumber` is not on the session type, query the user table:
```typescript
const userRecord = await db.query.user.findFirst({
  where: eq(user.id, session.user.id),
  columns: { phoneNumber: true },
});
if (!userRecord?.phoneNumber) {
  return NextResponse.json(
    { error: "Phone number required for callback" },
    { status: 400 }
  );
}
```

  <files>

  <action>
Modify the `human_detected` handler in the Vapi webhook to:

1. **Import** `orchestrateTransfer` from `@/lib/transfer-orchestrator` and `getStallInstructions` from `@/lib/stall-prompt`.
2. **Import** `user` from `@/db/schema` (for phone number lookup).

3. **In the `human_detected` handler** (inside the tool-calls loop), after the existing status update and event logging:

   a. Look up the user's phone number:
   ```typescript
   const userRecord = await db.query.user.findFirst({
     where: eq(user.id, callRecord.userId),
     columns: { phoneNumber: true, name: true },
   });
   ```

   b. Look up the user's name for the stall prompt. If `userRecord.name` is a placeholder (BetterAuth sets it to the email which is a temp email), use "the customer" as the name. Check if name contains "@" -- if so, use "the customer" instead.

   c. Get stall instructions: `const stallInstructions = getStallInstructions(userName);`

   d. **Fire-and-forget** the transfer orchestrator -- this MUST NOT block the webhook response (7.5s Vapi timeout). Use a void promise:
   ```typescript
   // Fire-and-forget -- do NOT await
   void orchestrateTransfer({
     callId: callRecord.id,
     vapiCallId: vapiCallId!,
     vapiControlUrl: callRecord.vapiControlUrl!,
     userPhone: userRecord.phoneNumber!,
   }).catch((err) => {
     console.error("Transfer orchestration failed:", err);
   });
   ```

   e. Return the stall instructions as the tool result instead of the current generic message:
   ```typescript
   results.push({
     toolCallId: toolCall.id,
     result: stallInstructions,
   });
   ```

4. **Guard the end-of-call-report handler**: When a Vapi call ends due to the AI being dropped (which we do intentionally during transfer), the `end-of-call-report` webhook fires. The current handler would overwrite the `connected` or `transferring` status with `completed`. Add a guard:
   - Before updating status in `end-of-call-report`, check the current call status.
   - If `callRecord.status` is `transferring` or `connected`, do NOT overwrite with `completed` or `failed`. Log the event but skip the status update. The conference-status webhook (from Plan 01) manages the final status for transferred calls.
   - Add this check:
   ```typescript
   const transferStatuses = ["transferring", "connected"];
   if (transferStatuses.includes(callRecord.status)) {
     // Call is in transfer flow -- Twilio conference webhooks manage status
     // Still log the event but don't change status
     await db.insert(callEvent).values({
       callId: callRecord.id,
       eventType,
       metadata: JSON.stringify({
         endedReason: endedReason ?? null,
         source: "end-of-call-report",
         skipped: "transfer-managed",
       }),
     });
     return new Response("OK", { status: 200 });
   }
   ```

5. Similarly guard the `status-update` handler for `ended` status: if the call is in `transferring` or `connected`, skip overwriting the status.
