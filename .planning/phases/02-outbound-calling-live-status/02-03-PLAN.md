  <action>
1. Create `src/app/call/[id]/page.tsx` (server component):
   - Import auth and db.
   - Get session via `auth.api.getSession({ headers: await headers() })`. Redirect to "/" if no session.
   - Extract `id` from params. Query call record with join on `isp` and `issueCategory` tables.
   - If call not found or call.userId !== session.user.id, redirect to "/select-isp".
   - Render `<CallStatusClient>` passing initial props: `callId`, `initialStatus`, `ispName`, `ispLogoUrl`, `categoryLabel`, `userNote`, `startedAt`, `endedAt`, `endedReason`.
   - This page should NOT use the step indicator. It is a different UX context (monitoring, not setup).

2. Create `src/app/call/[id]/call-status-client.tsx` ("use client"):
   - **Status display:** A clean, minimal card showing current status with a status label and optional description text. Use a step indicator showing call progression:
     - Steps: Dialing -> Ringing -> Connected -> On Hold -> Complete
     - Gray out future Phase 3 stages (Navigating Menu, Agent Found) -- define them in the status model but render them as disabled/hidden for now
     - Highlight current step. Mark completed steps with a check.
   - **Status mapping for display:**
     - "pending" -> "Starting..." / "Getting ready to place your call"
     - "dialing" -> "Dialing" / "Calling {ispName}..."
     - "on_hold" -> "On Hold" / "Waiting for an agent. We'll let you know when someone picks up."
     - "completed" -> "Complete" / show summary card
     - "failed" -> "Call Failed" / show error message with Try Again button
   - **Hold timer:** When status is "on_hold", show a subtle timer in the corner counting up from when the call connected (use `startedAt`). Format as "HH:MM:SS" if > 1 hour, "MM:SS" otherwise. Use `setInterval` with 1-second ticks. Minimal styling -- muted text, small font. Per user decision: "visible but not anxiety-inducing."
   - **Pusher subscription:** On mount, subscribe to `private-call-{callId}` using `getPusherClient()`. Bind to:
     - "status-change": `(data: { status: string, timestamp: string })` -> update status state
     - "call-ended": `(data: { status: string, endedReason: string, endedAt: string })` -> update to final state, show summary
   - **Connection state:** Bind to `pusher.connection.bind("state_change", ...)`. Track connected/disconnected. Show a small "Reconnecting..." banner at top when disconnected. Silent reconnect -- Pusher handles this automatically.
   - **Cancel button:** Always visible during active call states (pending, dialing, on_hold). On click, show `confirm("Are you sure you want to cancel this call?")` dialog (per user decision: "Are you sure?" confirmation prompt). If confirmed, POST to `/api/call/cancel` with `{ callId }`. Show loading state on button during request.
   - **Summary card (when call ends):** Clean receipt-style card showing:
     - Call result (based on endedReason -- use user-friendly labels from RESEARCH.md error messaging copy)
     - Duration (calculated from startedAt to endedAt)
     - ISP name and logo
     - Issue category
     - "Try Again" button that links back to `/confirm?isp={slug}&category={slug}&note={note}` (only show for failed calls)
     - "Start New Call" button that links to `/select-isp` (show for all ended calls)
   - **Error states with user-friendly messages:**
     - "customer-busy" -> "Line Busy" / "The ISP's line is busy right now."
     - "customer-did-not-answer" -> "No Answer" / "The call wasn't answered."
     - "exceeded-max-duration" -> "Call Timed Out" / "The call reached the maximum duration of 45 minutes."
     - "silence-timed-out" -> "Disconnected" / "The call was disconnected due to inactivity."
     - "manually-canceled" -> "Call Cancelled" / "You cancelled the call."
     - "twilio-failed-to-connect-call" -> "Connection Failed" / "We couldn't connect the call."
     - "assistant-error" -> "Something Went Wrong" / "An error occurred during the call."
     - Default/unknown -> "Call Ended" / "The call has ended."
   - **Cleanup:** On unmount, unsubscribe from Pusher channel, clear timer interval.
   - **Visual style:** Per user decision, "minimal & calm." Use Card from shadcn/ui. Subtle pulse animation on the status indicator while call is active (Tailwind `animate-pulse` on a small dot next to status text). No busy dashboard feel.

  <action>
1. Update `src/middleware.ts`:
   - Add `/call/:path*` to protected routes so unauthenticated users get redirected to "/".
   - Add active call redirect logic: When an authenticated user navigates to `/select-isp`, `/select-issue`, or `/confirm`, check if they have an active call. Fetch `GET /api/call/active` (using the request's cookies for auth). If `hasActiveCall: true`, redirect to `/call/{callId}`.
   - IMPORTANT: Do NOT check for active calls on `/call/*` routes (would cause redirect loop).
   - Do NOT check for active calls on `/`, `/verify`, or `/api/*` routes.
   - The active call check adds a fetch to middleware which adds latency. To mitigate: only check on the three ISP selection routes, not on every request.
   - Update the matcher to include `/call/:path*`:
     ```typescript
     export const config = {
       matcher: ["/select-isp", "/select-issue", "/confirm", "/call/:path*"],
     };
     ```

2. Update `src/app/layout.tsx`:
   - The StepIndicator should NOT render on `/call/*` pages. It's a setup flow indicator, not relevant during call monitoring.
   - Option A (simpler): Make StepIndicator conditionally render based on pathname. Since layout.tsx is a server component, use a wrapper client component that checks `usePathname()` and hides StepIndicator on `/call/*` routes.
   - Option B (route groups): Use Next.js route groups `(setup)` and `(call)` with separate layouts. This is cleaner but requires moving files.
   - **Choose Option A** for minimal disruption to existing file structure. Create a small wrapper or update the existing StepIndicator component to return null when pathname starts with "/call".
   - Update `src/components/step-indicator.tsx`: At the top of the component, check `if (pathname.startsWith("/call")) return null;`. This is the least-invasive change.
