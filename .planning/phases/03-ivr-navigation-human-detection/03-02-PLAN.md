Create `src/lib/prompt-builder.ts`:

Export a `buildNavigationPrompt` function that takes:
- `ispName: string` — e.g., "Comcast"
- `categoryLabel: string` — e.g., "Billing"
- `categorySlug: string` — e.g., "billing"
- `phoneTree: IspPhoneTreeConfig` — the ISP's phone tree data from the database

The function builds a system prompt string with these sections:

1. **Mission statement** — "You are an AI assistant calling {ispName} customer support. Navigate the phone tree to reach the {categoryLabel} department queue, then wait on hold for a live human agent."

2. **Phone tree map** — A formatted representation of the phone tree. Write a helper function `formatTreeForPrompt(nodes: PhoneTreeNode[], targetSlug: string, indent: number = 0)` that recursively formats the tree into readable text. For each node:
   - Show the node's prompt text (what the IVR says)
   - Show the action (press {keys} or say "{spokenResponse}")
   - If node has `categoryMatchSlugs` containing the targetSlug, mark it with ">>> TARGET PATH"
   - If node `isHoldQueue`, mark it with ">>> HOLD QUEUE - YOU ARE HERE, WAIT SILENTLY"
   - Recurse into children with increased indent

3. **Navigation rules** (per user decisions):
   - LISTEN to each menu prompt fully before pressing any keys
   - When you recognize a menu from the tree map, use the DTMF tool with the specified keys
   - PREFER pressing keys (DTMF) over speaking — more reliable
   - Insert pauses: use "w" (0.5s) between digits (e.g., "w1w2")
   - If menu doesn't match tree map, LISTEN carefully and navigate dynamically — choose option most likely to reach {categoryLabel}
   - If unsure, press 0 to try reaching an operator
   - If you hear "press star to repeat", press * and listen again
   - If asked for an account number, say "I don't have my account number available"
   - Do NOT provide any personal information

4. **Stuck recovery** (Claude's discretion — tiered strategy):
   - Tier 1: Press * to repeat current menu, listen again
   - Tier 2: Press 0 to try reaching a general operator
   - Tier 3: Call navigation_status with status "stuck", continue trying

5. **Hold behavior** (per user decisions):
   - Once in hold queue, WAIT SILENTLY
   - Do NOT speak during hold music or "please continue to hold" messages
   - Automated interruptions are noise — ignore completely
   - Include holdInterruptionPatterns from tree data as explicit examples to ignore

6. **Human detection** (per user decisions — conservative):
   - When a LIVE HUMAN AGENT answers (not a recording), call human_detected tool
   - Listen for: personal introduction ("Hi, my name is..."), direct questions to caller, conversational tone
   - Do NOT mistake hold interruptions for a human
   - Do NOT proactively say "Hello?" — wait for them to speak first
   - Be CONSERVATIVE — missed detection = wait a bit longer; false positive = wastes user's time

7. **Status reporting**:
   - Call navigation_status when entering a new menu level or reaching the hold queue
   - Use status values: "navigating" (in menu), "reached_queue" (found target queue), "on_hold" (waiting for agent), "stuck" (can't navigate)

If `navigationNotes` exists on the tree data, append it at the end as "ISP-Specific Notes: {notes}".

Expand the existing webhook handler in `src/app/api/webhooks/vapi/route.ts` to handle two new message types: `tool-calls` and `transcript`.

**tool-calls handler** — Add after the existing `status-update` handler:

```typescript
if (message.type === "tool-calls") {
  const vapiCallId = message.call?.id as string | undefined;
  if (!vapiCallId) return new Response("OK", { status: 200 });

  const callRecord = await db.query.call.findFirst({
    where: eq(call.vapiCallId, vapiCallId),
  });
  if (!callRecord) return new Response("OK", { status: 200 });

  const results = [];
  for (const toolCall of message.toolCallList ?? []) {
    if (toolCall.function?.name === "human_detected") {
      const params = toolCall.function?.arguments ? JSON.parse(toolCall.function.arguments) : {};

      // Update call status to agent_detected
      await db.update(call)
        .set({ status: "agent_detected", updatedAt: new Date() })
        .where(eq(call.id, callRecord.id));

      await db.insert(callEvent).values({
        callId: callRecord.id,
        eventType: "agent_detected",
        metadata: JSON.stringify({
          confidence: params.confidence,
          agentGreeting: params.agentGreeting,
        }),
      });

      // Push prominent alert to browser
      await pusherServer.trigger(callChannel(callRecord.id), "status-change", {
        status: "agent_detected",
        timestamp: new Date().toISOString(),
      });

      results.push({
        toolCallId: toolCall.id,
        result: "Human agent detected. Proceeding to transfer phase.",
      });
    } else if (toolCall.function?.name === "navigation_status") {
      const params = toolCall.function?.arguments ? JSON.parse(toolCall.function.arguments) : {};
      const navStatus = params.status;
      const detail = params.detail;

      // Map navigation status to call status
      const statusMap: Record<string, string> = {
        navigating: "navigating",
        reached_queue: "on_hold",
        on_hold: "on_hold",
        stuck: "navigating",
      };
      const newStatus = statusMap[navStatus] ?? "navigating";

      await db.update(call)
        .set({ status: newStatus as any, updatedAt: new Date() })
        .where(eq(call.id, callRecord.id));

      await db.insert(callEvent).values({
        callId: callRecord.id,
        eventType: "ivr_navigation",
        metadata: JSON.stringify({ status: navStatus, detail }),
      });

      await pusherServer.trigger(callChannel(callRecord.id), "status-change", {
        status: newStatus,
        detail,
        timestamp: new Date().toISOString(),
      });

      results.push({
        toolCallId: toolCall.id,
        result: "Status update received.",
      });
    }
  }

  return Response.json({ results });
}
```

IMPORTANT: The tool-calls webhook format from Vapi may vary. The research shows `toolCallList` as an array of objects, but the actual structure may have the function name at `toolCall.function.name` with arguments as a JSON string at `toolCall.function.arguments`, OR it may be at `toolCall.name` with `toolCall.parameters` as an object. Check the Vapi docs/SDK types if compilation fails and adjust accordingly. The webhook response format MUST be `{ results: [{ toolCallId, result }] }` where `result` is a string.

**transcript handler** — Add after the tool-calls handler:

```typescript
if (message.type === "transcript") {
  const vapiCallId = message.call?.id as string | undefined;
  const transcriptType = message.transcriptType; // "partial" or "final"
  const role = message.role; // "user" (ISP side) or "assistant" (our AI)
  const text = message.transcript;

  // Only forward final transcripts to avoid flooding Pusher
  if (!vapiCallId || transcriptType !== "final") {
    return new Response("OK", { status: 200 });
  }

  const callRecord = await db.query.call.findFirst({
    where: eq(call.vapiCallId, vapiCallId),
  });
  if (!callRecord) return new Response("OK", { status: 200 });

  // Push transcript to browser (no DB storage -- transient display data)
  await pusherServer.trigger(callChannel(callRecord.id), "transcript", {
    role,
    text,
    timestamp: new Date().toISOString(),
  });

  return new Response("OK", { status: 200 });
}
```
